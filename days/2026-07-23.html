注目事例と何が面白いかNotable examples and why they matter

今日の注目は PlayCanvas の SPZ/SOG ドキュメント群です。SPZ は Scaniverse/Niantic 系の持ち込み形式として扱いやすい一方、PlayCanvas docs は web delivery では SOG がより強いと明示します。SOG は画像ファイル群と metadata で構成されるため、表現面では『asset format の中身そのものが texture atlas』として見えます。これを単なるロード方式の違いで終わらせず、SPZ quantization、SOG LOD、glTF compatibility、GPU picking depth を色・fog・line・粒子密度に変換すると、編集ツールや鑑賞体験の中で形式の差が見えるようになります。

The notable source today is PlayCanvas' SPZ/SOG documentation set. SPZ is useful as a Scaniverse/Niantic ecosystem import format, while the PlayCanvas docs explicitly position SOG as stronger for web delivery. Because SOG is made from metadata plus image files, its internals already look like a texture atlas from an expression standpoint. Instead of treating this only as a loader difference, SPZ quantization, SOG LOD, glTF compatibility, and GPU-picking depth can become color, fog, line, and particle density inside authoring or viewing experiences.

表現・インタラクションのアイデアExpression and interaction ideas

中心案は『Format Pressure Scope』です。R は SPZ quantization / unsupported version risk、G は glTF/KHR compatibility、B は SOG LOD stream readiness、A は GPU picking depth confidence と置きます。ポインタは decode lens、長押しは streaming pressure、Space は 8K capture spike、F は format gate の切り替えです。7/22 の export resolution scope と合わせると、読み込み、編集、書き出しの各段階で同じ RGBA state を使えるようになります。短期では procedural tile shader で十分ですが、中長期では SOG の meta.json と image set、SPZ parser result、KHR_gaussian_splatting loader result を実データとして流し込みます。

The central idea is Format Pressure Scope. R represents SPZ quantization or unsupported-version risk, G represents glTF/KHR compatibility, B represents SOG LOD stream readiness, and A represents GPU-picking depth confidence. Pointer movement is the decode lens, holding adds streaming pressure, Space injects an 8K capture spike, and F toggles the format gate. Combined with the 07-22 export resolution scope, loading, editing, and exporting can share one RGBA state vocabulary. Short term, a procedural tile shader is enough. Mid/long term, SOG meta.json and image sets, SPZ parser results, and KHR_gaussian_splatting loader results should feed real data into the same visualization.

制作に使えそうな技術メモProduction notes

PlayCanvas の SPZ parser は engine 本体に直結した自動ロードというより、parser script と ZSTD WASM decoder を登録する設計です。この構造は『format capability を uniform 化する』のに向いています。SOG は means/scales/quats/sh0 などが WebP image set なので、thumbnail preview、chunk heatmap、missing channel overlay を shader で作りやすいです。Babylon.js 側は GaussianSplattingStream と GPUPicker の組み合わせが実装候補で、LOD switch/cooldown と picking confidence を別チャネルにできます。three.js r185 は HTMLTexture、TSL textureGather/storageTexture3D、compileAsync、render bundles が、注釈面・状態 texture・非同期準備表示に使えます。

PlayCanvas' SPZ parser is registered through a parser script plus a ZSTD WASM decoder rather than being an entirely implicit path, which makes format capability easy to turn into uniforms. SOG uses WebP image sets such as means, scales, quaternions, and base color/opacity, making thumbnail previews, chunk heatmaps, and missing-channel overlays natural shader tasks. On Babylon.js, GaussianSplattingStream plus GPUPicker is a practical target, with LOD switch/cooldown and picking confidence kept in separate channels. In three.js r185, HTMLTexture, TSL textureGather/storageTexture3D, compileAsync, and render bundles remain useful for annotation planes, state textures, and asynchronous readiness displays.

The recent trend is less about isolated shader looks and more about converting asset-pipeline uncertainty into shader-readable state. Since 07-17, SOG retry, fog/audio, upload queues, export preflight, and 8K/glTF scopes all belong to the same family: making time offsets and readiness visible.
