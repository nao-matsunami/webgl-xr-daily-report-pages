今日の重要トピックKey topics today

three.js r185 の HTMLTexture 対応、Babylon.js 9.14.0 の HtmlTexture、PlayCanvas v2.20.0 の copyElementImageToTexture と stereo XR gsplat が並び、HTML/DOM 系の情報面を 3D 空間に持ち込む下地が複数エンジンで揃ってきています。

three.js r185 adds HTMLTexture support, Babylon.js 9.14.0 adds HtmlTexture, and PlayCanvas v2.20.0 adds copyElementImageToTexture plus stereo XR gsplat. Multiple engines are converging on practical ways to bring HTML and DOM-derived information surfaces into 3D space.

注目事例と何が面白いかNotable example and why it matters

model-viewer の multi-model scene と Canvas/Video texture の例は、3D 本体と情報レイヤーを一つの空間で扱う設計の参考になります。特に『固定HUDではなく、意味レイヤーを空間に浮かせる』方向へつなぎやすいです。

The multi-model scene and Canvas/Video texture examples in model-viewer are useful references for treating the main 3D object and its information layers inside one shared space. They point toward floating semantic layers instead of a fixed HUD.

表現・インタラクションのアイデアExpression and interaction idea

常時読ませる UI ではなく、注目時だけ焦点面を合わせて情報が浮き上がる見せ方が有効です。普段は空間の気配として薄く存在し、視線やポインタ、会話の瞬間だけ読める層に切り替える構成が自然です。

Rather than making information readable at all times, it works better to let it rise into focus only when attention lands there. Most of the time it stays as a faint spatial presence, then becomes legible on gaze, pointer movement, or conversational cues.

制作に使えそうな技術メモProduction note

HTML-in-Canvas を前提にしておけば、Canvas2D や DOM ベースで組んだ意味レイヤーを、後から WebGL texture / postprocess / XR 投影へ拡張しやすいです。三次元側で全部を書き切るより、情報面の管理がかなり軽くなります。

If you assume HTML-in-Canvas from the start, meaning layers built in Canvas2D or DOM can later be extended into WebGL textures, post-processing, and XR projection more easily. It is much lighter than forcing every information layer to be authored directly in 3D.

試すならこれIf you try one thing

今日のサンプルは、複数の情報面を奥行きに置き、ポインタ上下で焦点面を移動させる最小構成です。読みやすさと演出を同じ素材で往復できるので、今後 WebGL 化して DOF 風処理へ伸ばす土台になります。

Today’s sample places multiple information surfaces in depth and moves the focal plane with vertical pointer motion. Because the same material can shift between readability and atmospheric staging, it is a good base for a future WebGL version with depth-of-field style processing.
